API: Application Programming Interface

서로 다른 소프트웨어 컴포넌트 간 통신하고 데이터를 교환할 수 있는 프로토콜

모던 웹 API는 2000년대 초반쯤 갖춰졌으며 약 다섯 단계를 거쳐 발전을 거듭했다

#### 1. 커머셜 API

2000년대 초반 웹 API는 신생 스타트업들이 상품을 만들 뿐만 아니라 온라인에서 판매할 수 있는 기반이 되었다

API는 기업 뿐만 아니라 해당 플랫폼을 이용하는 파트너와 리셀러까지 아울렀다

이 시기에 API는 주로 세일즈 포스, 이베이, 아마존에 의해 정의되었다

#### 2. 소셜 미디어 API

2000년대 중반 플리터, 페이스북, 트위터같은 SNS 기업들이 등장했다

이 시기 API는 이전 커머셜 API처럼 상품 판매 목적이 아닌 사용자의 정보를 공유하는데 이용되었다 

페이스북은 개발자들이 API를 통해 페이스북 유저의 친구, 사진, 프로필 정보에 접근할 수 있는 API를 2006년에 런칭했다

API가 직접적으로 수익을 가져다주지 않았지만 자신들의 플랫폼 영향력을 확대하는데 상당한 도움이 되었다

#### 3. 클라우드 API

아마존은 2006년에 S3와 EC2를 출시했는데, 이게 API의 또다른 전환점을 만들었다

두 서비스 모두 리소스에 접근하거나 인프라를 배포할 수 있는 API, CLI를 제공하여 개발자들이 편리하게, 비용 효율적으로 애플리케이션 개발을 가능하게 했다

#### 4. 모바일 앱에 대한 API

2007년에는 애플의 아이폰, 구글의 안드로이드가 출시되면서 웹은 모바일 기기에서 편하게 접근할 수 있게 되었고, 삶의 방식을 바꾸었다

이는 API 기반 모바일 앱에 대한 대규모 투자를 촉진했다

트윌리오는 API를 제품 형태로 제공하는 플랫폼을 런칭하여 개발자들이 모든 클라우드 애플리케이션에서 전화를 걸고 받을 수 있도록 했다

#### 5. 디바이스에 대한 API

2010년대에 들어서 개발자들은 모든 기기와 클라우드를 연결하기 위해 API를 사용하기 시작했다

카메라, 온도 조절기, 스피커, 마이크, 센서, 알렉사와 같은 스마트 스피커 등

즉, 인터넷에 접속할 수 있는 기기라면 API를 통해 데이터를 주고 받는 것이다

## API 구성 요소

API는 애플리케이션, 시스템, 장치 간 요청과 응답 사이클을 통해 데이터를 공유하며 동작한다

### 1. API 클라이언트

API 서버에게 요청을 전송함으로써 통신을 시작하는 컴포넌트

API 요청은 다양한 방식으로 트리거될 수 있다
- 검색창 입력
- 버튼 클릭
- 외부 이벤트(다른 애플리케이션으로부터 받은 알림 등)

### 2. API 요청

API 요청은 API 유형에 따라 모양과 동작 방식이 다르지만 일반적으로 아래의 정보들을 포함한다

**엔드포인트**
- 엔드포인트는 특정 리소스에 접근할 수 있는 URL을 의미한다
- 블로그 앱에서 `/articles` 엔드포인트는 아티클과 관련된 요청을 처리하는 로직을 포함할 수 있다

**메서드**
- 메서드는 클라이언트가 서버에게 전달한 리소스가 어떻게 처리되었으면 하는지 나타내는데 사용된다
- REST API는 표준 HTTP 메서드를 통해 접근 가능한데, 이 메서드에는 데이터 조회, 생성, 수정, 삭제와 같은 동작들이 포함된다

**파라미터**
- 파라미터는 리소스를 처리될 때 부가적인 정보를 제공하기 위해 엔드포인트에 전달되는 값이다
- 쿼리 스트링으로 API 요청 URL의 일부로 포함되거나 요청 본문에 담겨서 전달된다
- 블로그 앱에서 `/articles` 엔드포인트가 `topic` 파라미터를 가진다면, 특정 토픽에 대한 아티클만 조회하도록 파라미터를 이용할 수 있다

**요청 헤더**
- 요청 헤더는 key-value 쌍의 형태로 요청 자체에 대한 추가적인 디테일 정보를 제공한다
- 컨텐츠 타입이나 인증 정보 등

**요청 본문**
- 본문은 요청의 주요 부분으로 리소스를 생성, 수정, 삭제하는데 필요한 데이터를 담는다
- 블로그 앱에서 새로운 아티클을 만든다면 요청 본문에 제목, 내용, 작성자 등이 포함될 것이다

### 3. API 서버

API 클라이언트가 API 서버에 요청을 전달하면, API 서버는 인증 처리, 데이터 검증, 데이터를 조회 또는 조작하여 요청을 처리한 뒤 API 클라이언트에게 API 응답을 전달한다

### 4. API 응답

API 서버가 API 클라이언트에게 전달하는 응답에는 일반적으로 아래의 요소들이 포함된다

**상태 코드**
- HTTP 상태 코드는 세 자리수 코드 값으로 API 요청에 대한 결과를 나타낸다
- 200: 서버가 성공적으로 요청된 데이터를 반환했을 때
- 201: 서버가 새로운 리소스를 생성했을 때
- 404: 서버가 요청된 리소스를 찾지 못했을 때

**응답 헤더**
- 요청 헤더와 거의 유사하다
- 서버의 응답에 대한 추가적인 정보가 담겨있다

**응답 본문**
- 응답 본문은 API 클라이언트가 요청한 데이터를 보관한다
- 또는 문제가 발생했을 때 에러 메시지, 에러 코드를 보관한다


## API 유형

API에 접근할 수 있는 범위에 따라 크게 3가지 유형으로 분류한다

### 프라이빗 API

Private API = Internal API

조직 내의 서로 다른 소프트웨어 컴포넌트 간 통신하는데 사용되는 API로 로그인 처리 등, 내부 업무 용도로 쓰인다

서드파티 용으로 사용할 수 없다

### 퍼블릭 API

외부에서 조직의 데이터에 접근할 수 있게 해주는 API이다

무료이거나 비용을 지불해야 할 수 있다

예시: Stripe (결제 API 시스템)

### 파트너 API

완전 공개되지 않고 일부 조직들에서만 통신하기 위해 사용되는 API이다

인증된 파트너인지 보장하기 위한 인증 처리가 필요하다


## API 아키텍처

### 1. REST API

REST API는 API 엔드포인트를 통해 리소스에 접근할 수 있다고 취급한다

즉, 리소스를 URI 기반의 API 엔드포인트로 식별하고 표준 HTTP 메서드를 통해 데이터를 주고받는 API이다

### 2. SOAP

SOAP(Simple Object Access Protocol)은 클라이언트와 서버 간 완전히 구조화된 데이터를 주고 받기 위해 XML을 사용한다

보통 엔터프라이즈 환경이나 레거시 시스템에서 사용한다

조금 더 고급의 보안 기능을 포함하지만 다른 API 아키텍처에 비해 느리다는 단점이 있다

### 3. GraphQL

GraphQL은 클라이언트가 정확한 데이터를 얻기 위해 여러 번 API를 호출하지 않고 단일 API로 통신할 수 있게 해주는 오픈소스 쿼리 언어이다

클라이언트와 서버 간 왕복 횟수를 줄여주므로 느리거나 불안정한 네트워크 연결에서 실행되는 애플리케이션에 유용하다

### 4. Webhook

Webhook은 이벤트 기반 트리거에 대한 응답으로 자동적으로 요청을 전송하는데, 주로 이벤트 기반 아키텍처를 구현할 때 사용된다

결제같은 이벤트가 발생하면 애플리케이션은 미리 설정된 웹 훅 URL에 관련된 데이터를 요청 페이로드에 담아  HTTP 요청을 보낸다

웹 훅을 받는 시스템은 이벤트 발생에 대한 요청을 처리하여 적절한 액션을 취할 수 있다

### 5. gRPC

gRPC (Remote Procedure Call)는 클라이언트가 로컬 객체처럼 호출하여 분산 애플리케이션과 시스템 간의 통신을 용이하게 한다


## API 끝 슬래시

`/users`와 `/users/` API 엔드포인트는 서버 입장에서 다른 URL로 취급될 수 있다

Apache, Nginx에서는 `/users/`를 디렉토리로, `/users`를 파일로 간주한다

REST API에서는 `/users`는 사용자 목록(컬렉션), `/users/123`은 특정 사용자(리소스)를 나타낸다 -> 팀에서 일관성있게 표시하는 게 중요함

대부분 프레임워크는 다음 중 하나를 선택한다
- 자동 리다이렉트: `/users/`로 요청받으면 `/users`로 리다이렉트 (또는 반대) - 301/308
- 둘 다 허용하지만 내부적으로 한 가지 URL로 정규화
- 하나만 허용하고 나머지 하나는 404 처리

스프링 부트: 끝 슬래시 없음 `/users`

장고: 끝 슬래시 있음 `/users/`

## API First

기존 개발 과정은 요구사항 분석 -> 화면 설계 -> 백엔드 API 구현 -> 프론트 구현 -> 테스트 순으로 이어졌다

이 방식은 백엔드의 API 구현이 늦어지면 프론트의 구현 기간에 영향을 준다

백엔드에서 기한 내에 API를 구현하더라도 프론트에서 API를 수정해야 한다면 백엔드에서 다시 수정을 해야 하므로 개발 기간이 늦어질 수 있다

API First 방식은 API 설계를 앞 단에서 진행하는 방식으로, 개발자들이 함께 API 설계에 참여하고 백엔드와 프론트가 비즈니스 로직을 동시에 구현할 수 있게 한다
