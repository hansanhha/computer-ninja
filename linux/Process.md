## 프로세스

시스템 상에서 실행되는 프로그램을 프로세스라고 한다.

사용자가 명령어를 입력하면 운영체제는 명령어를 실행하기 위해 해당 명령어가 위치한 파일 경로에 위치한 바이너리를 실행하여 프로세스를 생성한다.

프로그램: `/bin/bash`, `/usr/bin/python`

프로세스: 메모리 위에서 실행 중인 인스턴스 (같은 프로그램을 여러 번 실행하면 여러 프로세스가 만들어짐)

프로세스는 프로그램을 실행하기 위한 정보인 '실행 컨텍스트'를 가지고 있다.

```text
PID
메모리 공간
코드
힙
스택
CPU 레지스터 상태
열린 FD
환경 변수
현재 작업 디렉토리
signal 상태 등등
```

PID는 시스템 내에서 프로세스를 구분하고 추적/관리하기 위해 운영체제가 부여한 고유의 5글자짜리 ID 값이다.

이 값은 절대 중복될 수 없으며 (프로세스가 종료된 후 동일한 ID 값이 나중에 사용될 수는 있음), 시스템은 이 ID 값을 통해 각 프로세스를 추적하거나 관리할 수 있다.


## 프로세스 생명주기

프로세스 생명주기는 일반적으로 아래와 같은 상태를 거치게 된다.

`생성(New)` -> `준비(Ready)` -> `실행 (Running)` -> `대기/보류(Waiting/Blocked)` -> `종료(Terminated/Exit)`

생성: 프로그램이 메모리에 적재되고 프로세스가 생성되는 단계 (PCB 할당)

준비: 프로세스가 CPU 스케줄러로부터 CPU를 할당받기 위해 대기하는 상태

실행: 프로세스가 CPU 스케줄러로부터 CPU를 할당받은 뒤 CPU 명령어를 실제로 수행하는 상태

대기/보류: 프로세스가 입출력 작업이나 이벤트 발생을 기다리며 CPU 사용을 멈추고 대기하는 상태 (CPU를 일시적으로 반납하고 해당 작업이 완료되면 '준비' 상태로 돌아감)

종료: 프로세스 실행이 완료되거나 오류로 인해 강제 종료된 상태 (메모리 반환 및 PCB 삭제)


## `fork()`, `exec()`

`fork()` (프로세스 복제) -> `exec()` (프로세스 실행)

`fork()` 함수는 새로운 프로세스를 생성하기 위해 메모리를 할당하고 현재 실행 컨텍스트를 복제하는 시스템 콜이다. 

fork를 하게 되면 부모-자식 두 프로세스가 만들어지며 자식은 부모 컨텍스트를 그대로 물려받기 때문에 부모 프로세스에서 실행 중이던 다음 코드를 이어서 실행하게 된다. (부모-자식 프로세스가 유일하게 다른 건 PID 값)

`exec()` 함수는 fork와 달리 새 프로세스를 만들지 않고 현재 프로세스를 새 프로그램으로 교체하는 시스템 콜이다. (exec는 여러가지 함수로 구성됨)

PID는 유지하면서 메모리 공간, 코드, 힙, 스택을 전부 새 프로그램의 상태로 덮어쓴다.

따라서 `exec`를 호출한 이후의 부모 프로세스 코드는 실행되지 않고 `exec`로 호출된 명령어만 실행된다.

터미널에서 `grep hello file.txt` 명령어를 실행하면 쉘은 아래와 같은 순서를 진행한다.

```text
1. shell 자신을 복제 (fork) - 부모 bash -> 자식 bash clone
2. 자식 프로세스에서 grep 프로그램 실행 (exec)
3. 부모 shell은 자식 쉘의 프로세스가 끝나길 기다림 (wait)
```

fork-exec 구조 덕분에 부모 프로세스는 `exec` 전에 자식 프로세스를 원하는 대로 설정할 수 있다.

리다이렉션이나 파이프 등을 사용하여 입/출력의 대상을 변경할 수 있는 것은 이러한 메커니즘 덕분이다.

```shell
# 리다이렉션
ls > out.txt

# 1. fork() -> 자식 생성
# 2. 자식 stdout(fd=1) 변경
# 3. exec(ls) 자식 프로세스 실행
```

```shell
# 파이프
cat app.log | grep hello

# 1. pipe 생성
# 2. fork(cat)
# 3. fork(grep)

# 4. cat stdout(fd=1) 변경 -> pipe write
# 5. grep stdin(fd=0) 변경 -> pipe read
```

## 포그라운드/백그라운드 프로세스

프로세는 크게 **포그라운드 프로세스**와 **백그라운드 프로세스**로 나뉜다.

기본적으로 모든 프로세스는 **포그라운드 프로세스(Foreground process)**로 실행된다.

포그라운드 프로세스는 키보드로부터 입력을 받고 터미널로 출력값을 전송할 수 있다.

다만 프로세스가 종료될 때까지 터미널을 블로킹하게 된다.

이와 달리 **백그라운드 프로세스(Background process)**는 터미널과 독립적으로 실행되어 다른 작업을 병행할 수 있다. (실행되는 동안 키보드 입력을 필요로 하지 않기도 함)

```shell
# 명령어가 가장 마지막에 '&'를 붙여서 백그라운드 프로세스로 실행할 수 있다
sleep 10 &
```

## 좀비, 고아, 데몬 프로세스

**좀비 프로세스(zombie process)**: 자식 프로세스의 실행이 종료됐지만 부모 프로세스가 아직 회수하지 않은 프로세스

```text
bash
 └─ python app.py
```

bash: 부모 프로세스

python: 자식 프로세스

자식 프로세스가 종료되면 커널은 곧바로 삭제하지 않고 종료 코드(Exit status), CPU 사용량, 종료 원인(Signal 여부) 등을 읽어야 하기 때문에 PID, 종료 상태와 같은 최소한의 정보만 남겨둔다.

좀비 프로세스는 실행되지 않으며 종료 정보 보관용으로만 사용된다. (CPU 사용 X)

부모 프로세스는 자식 프로세스가 종료되면 `SIGCHLD` 시그널을 받는데 `wait()`를 통해서 자식 프로세스를 회수할 수 있다.

만약 wait 하지 않으면 좀비 프로세스인 상태로 계속 유지된다.

**고아 프로세스(orhan process)**: 부모 프로세스가 종료되었으나 자식 프로세스가 실행 중인 상태

부모 프로세스가 먼저 종료되는 경우 `init` 프로세스가 자식 프로세스의 새로운 부모 프로세스로 설정된다. (**orphan adoption**)

`init` 프로세스는 컴퓨터가 부팅될 때 생성되는 최초의 프로세스로, 컴퓨터가 종료될 때까지 실행되는 데몬 프로세스이다.  (PID = 1)

고아 프로세스의 실행이 종료되면 `init` 프로세스가 wait를 호출하여 고아 프로세스의 종료 상태를 회수한다.

**데몬 프로세스(deamon process)**: 사용자와 상호작용하지 않고 백그라운드에서 지속적으로 동작하는 서비스 프로세스

파이썬 프로세스를 하나 실행해본다고 가정해보면 아래와 같은 상태가 된다.

```shell
$ python app.py

Terminal
  ↓
zsh/bash
  ↓
python
```

파이썬 프로세스는 부모 프로세스(shell)가 있고 터미널(stdin, stdout, stderr)에 연결된다.

또한 사용자가 로그아웃하면 영향을 받을 수 있다.

데몬 프로세스의 데몬은 '악마'가 아니라 눈에 보이지 않게 뒤에서 일을 처리하는 존재를 의미한다.

그래서 이러한 프로세스의 이름에는 `d`가 붙어서 데몬 프로세스임을 나타낸다. (sshd, crond 등)

데몬은 터미널과 독립적으로 실행되며 된다. (터미널로부터 분리되어 독립적인 세션을 보유함)

그리고 PID 1(systemd/lancud)의 관리를 받는다.

```text
// 일반 프로세스
Terminal
 ↓
Shell
 ↓
Program

// 데몬 프로세스
systemd/launchd
 ↓
daemon
```

백그라운드 프로세스라고 해서 데몬 프로세스가 되지 않는다.

백그라운드 프로세스는 여전히 shell의 자식이며 독립적인 세션을 보유하지 않기 때문이다.

```shell
python app.py & # 백그라운드 프로세스
```

데몬 프로세스를 터미널과 분리하는 이유가 뭘까?

ssh로 서버를 접속한 후 프로세스를 실행했을 때, 해당 프로세스가 터미널에 묶여 있다면 ssh 접속을 종료함과 동시에 프로세스가 종료될 수 있다.

서비스를 사용자의 로그인 여부와 무관하게 동작시키기 위해 터미널과 분리하는 것이다.

```shell
# 서버 접속
ssh server

# 프로세스 실행
./myserver

# Ctrl+C, Ctrl+Z, ssh 접속 종료 -> 프로세스 종료
exit
```

## 프로세스 그룹, 세션

```text
Session
 └─ Process Group
      └─ Process

Session = 회사
Process Group = 팀
Process = 직원
```

```text
TTY는 세션에 연결되어 포그라운드 프로세스 그룹을 기억하다가 Ctrl+C, Ctrl+Z 같은 신호를 해당 그룹 전체에 전달한다
```

**프로세스 그룹**이란 하나 이상의 프로세스를 묶어서 하나의 작업처럼 다루기 위한 단위이다.

아래의 `cat`과 `grep`은 별도의 프로세스로 동작하지만 사용자 입장에서는 하나의 명령이다.

```shell
cat file.txt | grep hello
```

그래서 쉘은 프로세스 그룹을 만든다.

```text
Process Group #100
 ├─ cat   (PID 100)
 └─ grep  (PID 101)
```

모든 프로세스는 PID 뿐만 아니라 PGID (Process Group ID)를 가진다.

PGID의 값은 보통 그룹 리더의 PID 값으로 설정된다.

```text
PID   PGID
100   100
101   100
```

`Ctrl + C`를 누르면 프로세스 그룹에 SIGINT를 전달하여 모든 프로세스가 종료된다.

터미널은 항상 포그라운드 프로세스 그룹을 가지는데, 현재 사용자의 입력이 모두 이 그룹에 전달된다.

```text
Ctrl+C
↓
TTY
↓
Foreground Process Group
↓
SIGINT 전송
```

**세션**은 프로세스 그룹보다 더 큰 단위로 여러 프로세스 그룹을 묶는 상위 컨테이너이다.

보통 bash, sshd, login 등의 프로세스로 인해 만들어진다.

```text
Session
 ├─ Process Group A
 ├─ Process Group B
 └─ Process Group C 
```

세션은 하나의 터미널을 소유할 수 있다.

사용자가 터미널에 접속한 후 vim을 실행하면 아래와 같은 구조가 된다.

```text
Session
 ├─ bash group
 └─ vim group
```

데몬 프로세스는 일반 프로세스와 달리 터미널이 세션에 소유되지 않고 독립적으로 존재하여 `Ctrl + C`나 로그아웃 영향을 받지 않는다.

```text
// 일반 프로세스
Terminal
 ↓
Session
 ↓
Shell
 ↓
Program

// 데몬 프로세스
Session
 ↓
Daemon
```
