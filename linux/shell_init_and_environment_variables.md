## 시스템 부팅 과정

전원 버튼을 누르면 UEFI 펌웨어/BIOS가 실행되고 UEFI는 OS를 탐색하는 부트로더(GRUB 등)를 실행시킨다.

부트로더는 커널을 메모리에 로드한 뒤 CPU 제어권을 커널에 넘긴다.

커널은 메모리, CPU, 스케줄러, 파일시스템, 디바이스 드라이버 등을 초기화하여 운영체제를 시작한다.

그리고 모든 프로세스의 루트이자 PID 1번인 systemd/launchd 프로세스를 생성한다.

PID 1은 여러 서비스들을 실행한다.

```text
systemd
 ├─ sshd
 ├─ cron
 ├─ nginx
 ├─ dbus
 ├─ login manager
 └─ ...
```

이후 로그인에 성공하면 사용자 정보를 조회하여 사용자 환경/세션을 생성하고 로그인 쉘을 실행한다.

```terminal
Power On
   ↓
UEFI
   ↓
GRUB
   ↓
Kernel
   ↓
systemd/launchd (PID 1)
   ↓
getty
   ↓
login
   ↓
사용자 인증
   ↓
setsid()
   ↓
TTY 연결
   ↓
exec(bash)
   ↓
Shell Prompt
```


## 쉘 초기화 파일

쉘은 사용자의 입력을 해석하고 코드를 실행하여 운영체제에 전달하는 프로그램이다.

쉘 프로세스는 생성될 때 쉘 초기화 파일을 실행시켜 사용자 코드(쉘 스크립트)를 통해 쉘 환경을 설정할 수 있게 한다.

쉘 초기화 파일은 쉘이 Homebrew, Go 등 각종 라이브러리와 도구들을 실행할 수 있도록 경로를 설정하거나 명령어 별칭 지정, 프롬프트 구성, 플러그인 로딩을 위해 사용된다.

**쉘 초기화 파일에 이러한 쉘 스크립트를 작성해두면 쉘이 생성될 때마다 사용자가 일일이 타이핑하지 않고도 자동으로 로그인 환경을 재현할 수 있다.**

이 때 쉘의 타입에 따라 실행되는 쉘 초기화 파일이 다른데, 쉘은 크게 로그인 쉘/논-로그인 쉘, 인터랙티브 쉘/논-인터랙티브 쉘로 나뉜다.
- 로그인 쉘: 사용자 로그인 과정에서 시작된 쉘
- 논-로그인 쉘: 이미 실행 중인 쉘에서 새로 실행된 쉘
- 인터랙티브 쉘: 프롬프트를 통해 사용자가 쉘과 소통할 수 있는 쉘
- 논-인터랙티브 쉘: 프롬프트가 없는 스크립트 실행용 쉘

```text
Kernel
 ↓
Session
 ↓
TTY
 ↓
Shell
 ↓
Shell File (Shell Init Code)
```

### zsh 쉘 파일 시작 순서

쉘의 종류에 상관없이 모든 zsh는 가장 먼저 아래의 경로에 놓인 파일을 읽어서 **PATH, 환경변수를 설정**한다.

```text
/etc/zshenv
~/.zshenv
```

그 다음 **로그인 쉘**인 경우 아래의 파일을 읽어 **PATH 설정과 로그인 초기화 과정을 수행**한다.

```text
/etc/zprofile
~/.zprofile
```

**인터랙티브 쉘**인 경우 아래 경로의 파일을 읽어서 나머지 설정을 실행한다.

```text
/etc/zshrc
~/.zshrc
```

**로그인 + 인터랙티브 쉘**인 경우 아래의 순서로 파일이 실행된다.

```text
.zshenv
   ↓
.zprofile
   ↓
.zshrc
   ↓
.zlogin
```

쉘에서 `zsh`를 입력하여 쉘을 또 실행하면 논-로그인 쉘이므로 `.zprofile` 파일은 실행되지 않는다.

```text
.zshenv
   ↓
.zshrc
```

`zsh script.sh`와 같이 스크립트를 실행하면 논-로그인 쉘, 논-인터랙티브 쉘이므로 `.zshenv` 파일만 실행된다.

### 쉘 초기화 파일 작성

```shell
# 환경변수 설정
export EDITOR=vim
export JAVA_HOME=/Library/Java/JavaVirtualMachines/jdk-21.jdk/Contents/Home

# PATH 설정
export PATH="$HOME/bin:$PATH"

# 명령어 별칭 지정
alias gs='git status'

# 플러그인 로딩
source ~/.fzf.zsh
```


## 환경변수

자바 프로그램을 실행한다고 가정해보자.

자바는 실행되기 위해 다음 정보를 필요로 할 수 있다.

```text
사용할 언어 (영어, 한국어 등)
시간대
홈 디렉토리
설정 파일 위치
```

이런 정보를 프로그램을 실행할 때마다 직접 입력하면 불편하므로 **'프로세스가 실행될 때 설정값을 함께 전달하기 위해'** 환경변수를 사용한다.

환경변수는 쉘에서만 사용하는게 아니라 GUI를 포함한 모든 프로그램에서 사용된다.

그리고 **커널은 환경변수에 대해 직접 관여를 하지 않는다.** -> 운영체제는 환경변수를 모른다.

**환경변수는 부모 프로세스에서 자식 프로세스로 상속**되는데, 시스템 부팅 시 최초로 실행되는 프로세스인 systemd/launchd 프로세스에서 `/etc/profile` 등의 파일을 읽어서 초기 환경변수를 구성한다. (리눅스의 경우 `/etc/environment`)

쉘의 경우 쉘 초기화 파일이 실행되면서 환경변수가 자동으로 구성되지만 GUI 프로그램은 쉘 초기화 파일을 읽지 않으므로 PATH 값이 다르게 되어 해당 프로그램에서 별도로 설정해야 한다는 차이점이 있다.

환경변수는 `export env_name=env_value` 형태로 지정할 수 있다.

일반적으로 사용자별 환경변수는 로그인 파일(`~/.zshrc`, `~/.bashrc`)에 명시하여 설정한다.

```shell
export CONFIG=$HOME/.config
```

### 환경변수 명령어

`env`, `printenv`

`export`

`setenv`, `unsetenv`

`which`, `where`, `command -v`


### PATH 환경변수

PATH 환경변수는 쉘이 명령어 실행 파일을 찾기 위해 검색하는 디렉토리 목록이다.

터미널에서 `ls`를 입력하면 쉘은 `ls` 라는 이름만 받는다.

이를 실행하려면 `ls`라는 이름을 가진 프로그램의 파일 경로가 필요하다.

쉘은 `ls` 프로그램이 위치한 `/bin/ls`을 찾아서 실행한 뒤 결과를 표시한다.

이 때 프로그램의 파일 경로를 PATH 환경변수를 통해 찾는다.

PATH 환경변수에는 실행 파일이 놓인 디렉토리 목록이 다음과 같은 형식으로 나열되어 있다.

```text
/usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin
```

쉘은 `:`으로 구분된 디렉토리 리스트를 순서대로 검색하여 사용자가 입력한 프로그램 이름과 동일한 파일이 있는지 검색한다. 

```text
/usr/local/bin/ls
     ↓ 없음
/usr/bin/ls
     ↓ 발견
exec("/usr/bin/ls")
```

### 환경변수 vs 설정파일

환경변수와 설정파일은 둘 다 프로그램에 설정을 전달하는 방법이지만 사용되는 목적이 다르다.

환경변수는 **프로세스를 실행하기 위한 실행 환경(Runtime Envrionment)**을 전달한다.

보통 배포 환경마다 달라지는 값을 저장한다. (DB 주소, API 키, 포트 번호, 활성 프로파일(prod/dev), 외부 서비스 주소)

설정파일은 **애플리케이션이 동작하기 위한 설정(Configuration)**을 저장한다.

애플리케이션 자체 설정 값들을 저장한다. (로깅 설정, 캐시 정책, 비즈니스 규칙, 기능 옵션)
